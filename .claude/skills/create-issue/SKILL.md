GitHub Issue を作成・管理する際のガイドライン。

## ラベル体系

### カテゴリラベル（必ず1つ以上付与）

| ラベル | 用途 | 例 |
|--------|------|----|
| `bug` | 仕様と異なる動作、クラッシュ | 猫又道連れで背徳者が後追いしない |
| `enhancement` | 新機能の追加、既存機能の改善 | ゲームログ機能、ルール説明画面 |
| `game-rule` | ゲームルール・メカニクスに関する変更 | 連続ガード禁止、決選投票、仲間認識設定 |
| `UI/UX` | 画面デザイン・操作性・演出 | アニメーション、iPad対応、共有形式改善 |
| `documentation` | ドキュメントの追加・改善 | README更新 |

- 複数カテゴリに該当する場合は複数付与してよい（例: `bug` + `game-rule`）
- `game-rule` はゲームの進行ロジックや勝敗条件に影響するものに付ける
- 純粋なUI変更で game-rule に影響しないものは `UI/UX` のみ

### 優先度ラベル（必ず1つ付与）

| ラベル | 基準 |
|--------|------|
| `priority: high` | ゲームが正しく動作しない（バグ）、または機能の根本的な欠損 |
| `priority: medium` | ゲームプレイの質を向上させるが、なくても遊べる |
| `priority: low` | あると嬉しいが、余裕があれば対応（演出、最適化など） |

**優先度の判断基準:**
- バグは原則 `high`。ただし極めてレアケースのみ発生するバグは `medium` でもよい
- ハウスルールの追加オプションは `medium`（一般的に使われるもの）または `low`（ニッチなもの）
- UI演出・見た目の改善は `low`
- 初心者体験に直結するもの（ヘルプ、プリセット等）は `medium`

## Issue 本文の構成

### バグの場合
```markdown
## バグ概要
（1-2文で現象を説明）

## 再現手順
1. ...
2. ...

## 期待される動作 / 実際の動作

## 原因
（コードレベルでの原因。ファイル名・行番号を含める）

## 修正方針
（どう直すかの概要）
```

### 機能追加の場合
```markdown
## 概要
（何を追加するか）

## 背景
（なぜ必要か、どういうゲーム体験の改善か）

## 提案
（具体的な仕様・UI案）

## 関連Issue
（あれば）
```

## Issue 作成時の注意

- タイトルは具体的に。「○○の改善」ではなく「○○した時に△△が起きない」のように書く
- 1つのIssueに複数の独立した変更を混ぜない
- 既存Issueと重複しないか `gh issue list` で確認してから作成する
- 関連するIssueがあれば本文中で `#番号` で参照する
- `gh issue create --title "..." --body "..." --label "label1,label2"` でラベルも同時に付与する
